## 2. 角色视角的工作清单（按你是谁勾选）

### 产品经理（PM）
- [ ] **需求澄清**：把需求写成“可测试的行为”，补齐边界与反例
- [ ] **验收标准**：写出“通过/不通过”的清晰条件（最好 5 条以内）
- [ ] **拆分任务**：拆成前端/后端/运营/数据/设计可并行的小块
- [ ] **风险与假设**：列出 1–3 个关键假设 + 如何验证
- [ ] **跟进节奏**：今天要在哪个时间点收敛一次结论

### 前端（FE）
- [ ] **页面/交互范围**：明确影响到的页面、状态、埋点
- [ ] **接口契约**：对齐请求/响应字段、错误码、加载/空/失败态
- [ ] **可验收产出**：至少一个可点击的原型/截图/录屏/预览链接
- [ ] **测试/回归点**：关键路径 + 1 个最容易坏的边界条件

### 后端（BE）
- [ ] **数据与权限**：数据模型/权限校验/审计字段是否明确
- [ ] **接口与兼容**：接口契约、幂等、分页、限流、错误处理
- [ ] **可观测性**：日志/指标/告警（至少能定位失败原因）
- [ ] **测试/回归点**：核心逻辑单测 + 关键集成路径

### 运营（Ops/增长/内容）
- [ ] **发布物料**：文案/图片/FAQ/公告/社群话术
- [ ] **上线节奏**：灰度策略、触达名单、时间点、回滚预案
- [ ] **指标定义**：今天要看什么指标判断“有效/无效”
- [ ] **反馈闭环**：收集渠道、分级规则、回复模板

## 3. 用 AI 当队友（把“说清楚”当作工作的一部分）

> 建议每天至少做一次：让 AI 产出**计划/对比方案/风险清单/测试用例**之一，然后你来拍板。

- [ ] **最佳 N 方案**：让 AI 给 3–4 个实现/方案，并说明取舍
- [ ] **计划文件**：让 AI 输出“今天要改哪些文件/要做哪些验证”
- [ ] **对齐契约**：把接口契约/埋点表/验收标准让 AI 复述一遍，确认无歧义
- [ ] **自检清单**：让 AI 用 `REVIEW_CHECKLIST.md` 自己审一次

可直接复制给 AI 的提示词（任选其一）：

```text
请把下面目标拆成一个“今天可完成”的执行计划。要求：
1) 列出要创建/修改的具体文件（若是非代码任务则列出具体产出物）
2) 每一步的验收方式（截图/日志/测试/链接）
3) 风险与假设（最多 3 条）以及验证方法
目标：{粘贴你的目标}
上下文：{粘贴需求/链接/接口约定}
```

```text
请基于下面的验收标准，生成 8–12 条测试用例（覆盖正常/异常/边界），并指出哪些用例更适合前端测、后端测、联调测。
验收标准：{粘贴}
接口契约（如有）：{粘贴}
```
